# 深度对比报告：Casdoor vs Logto

> **摘要**：针对您的个人服务器环境（腾讯云广州、4核4G、备案需求）及“麻雀虽小五脏俱全”的诉求，本报告对 **Casdoor** 与 **Logto** 进行了深度对比分析。
>
> **结论先行**：
> *   **首选 Casdoor**：如果您追求极致的资源效率、最全的免费功能、以及最彻底的国内本地化支持。
> *   **备选 Logto**：如果您是前端颜值控，愿意牺牲一定的服务器资源换取更现代化的 UI/UX 交互体验。

---

## 1. 产品定位与核心差异

| 维度 | **Casdoor** | **Logto** |
| :--- | :--- | :--- |
| **一句话定位** | **功能优先**的轻量级 IAM 平台，主打 RBAC/ABAC 强权限管理。 | **体验优先**的现代化 CIAM 平台，主打开发者体验与绝美的 UI 设计。 |
| **核心技术栈** | **Go** (后端) + React (前端) | **Node.js** (后端) + React (前端) |
| **维护团队** | Casbin 社区 (国内顶尖安全中间件团队) | Silverhand Inc. (国际化团队) |
| **开源策略** | **全功能开源**，无明显的“企业版”功能锁。 | 核心开源，部分高级协作功能（如多管理员）可能受限或属于云版特性。 |

## 2. 深度维度分析

### 2.1 资源消耗与架构轻量度
对于单机部署（特别是个人服务器），资源占用是核心考量。

*   **Casdoor (胜出 🏆)**
    *   **架构**：Go 语言编译为二进制文件，天然轻量。
    *   **内存**：空闲状态下仅需 **~50MB - 100MB**。
    *   **启动速度**：毫秒级启动。
    *   **数据库**：原生支持 **MySQL**（您服务器上可能已有），也支持 Postgres/SQL Server 等。
    *   **评价**：真正的“麻雀”，对服务器几乎无负担，能留出更多资源给您的业务应用。

*   **Logto**
    *   **架构**：基于 Node.js 微服务架构，依赖较多 npm 包。
    *   **内存**：通常起步需 **500MB+**，负载高时波动较大。
    *   **数据库**：强依赖 **PostgreSQL**。如果您的服务器主要运行 MySQL，引入 Postgres 会增加额外的运维负担（多一个数据库进程）。
    *   **评价**：虽然不算重型（比 Keycloak 轻），但相比 Casdoor 仍显得“富态”。

### 2.2 国内环境适应性（备案与合规）
这是您最关心的痛点之一。

*   **Casdoor (胜出 🏆)**
    *   **社交登录**：原生支持微信、QQ、钉钉、企业微信、飞书、微博等几乎所有国内主流平台，配置极其详细。
    *   **CDN 与网络**：静态资源无境外依赖，完全适配国内网络环境，加载速度快。
    *   **中文支持**：文档由中文母语团队编写，沟通与问题排查无语言障碍。
    *   **短信/邮件**：内置阿里云、腾讯云、七牛云等国内厂商的短信/邮件发送接口，开箱即用。

*   **Logto**
    *   **社交登录**：支持微信、支付宝等，但国内长尾平台支持不如 Casdoor 全面（主要通过 Connector 扩展）。
    *   **网络**：部分 Connector 或静态资源可能依赖境外服务，偶尔可能遇到网络波动（虽然已有很大改善）。
    *   **本地化**：虽然支持中文，但核心逻辑更偏向国际标准（Auth0 风格），对国内特有的“验证码登录”等习惯支持度略有差异。

### 2.3 功能完备性与权限模型

*   **Casdoor (胜出 🏆)**
    *   **权限管理**：这是 Casdoor 的杀手锏。它集成了 Casbin 强大的权限引擎，支持 RBAC（角色访问控制）、ABAC（属性访问控制）。
    *   **多租户**：原生支持多组织（Organization）隔离，您可以在一个实例里为不同的圈子朋友开不同的“租户”。
    *   **前端集成**：提供了极其丰富的 SDK（React, Vue, Angular, Android, iOS, Go, Java, Python...）。

*   **Logto**
    *   **权限管理**：支持基础的 RBAC（Role-based），对于大多数应用足够，但不如 Casbin 灵活。
    *   **机器交互 (M2M)**：支持良好，适合微服务调用。
    *   **UI 定制**：提供了非常易用的“体验配置”，可以无代码修改登录页品牌元素，比 Casdoor 的配置更直观。

### 2.4 开发者体验 (DX) 与 UI 颜值

*   **Logto (胜出 🏆)**
    *   **UI 设计**：Logto 的管理后台和默认登录页设计具有国际一流水准，简洁、现代、流畅。
    *   **引导流程**：拥有“保姆级”的接入引导，几乎零门槛上手。
    *   **文档**：文档结构清晰，示例丰富，非常适合前端开发者。

*   **Casdoor**
    *   **UI 设计**：偏向工程师风格，功能堆叠较多，实用但不够“潮”。
    *   **管理后台**：选项极其丰富，初学者可能会被满屏的配置项（组织、应用、提供商、资源、权限、模型...）吓一跳。
    *   **上手难度**：需要理解一定的 IAM 概念（如 Realm/Organization 的区别）。

## 4. 针对您情况的最终建议

### 您的场景画像：
*   **硬件**：腾讯云 4核 4G（资源宝贵）。
*   **网络**：国内备案环境（需稳定连接）。
*   **需求**：个人使用，功能要全，不要太复杂（运维层面）。
*   **现状**：可能已有 MySQL。

### 最终推荐：Casdoor

**核心理由**：
1.  **省资源**：4G 内存跑 Java/Node 应用多了会捉襟见肘，但跑 Go 应用绰绰有余。Casdoor 极其克制，不会抢占您核心业务的资源。
2.  **省运维**：它支持 MySQL。如果您服务器上已经跑了 MySQL，您甚至不需要起新的数据库容器，直接复用即可。而 Logto 强行要求您维护一个 Postgres，增加了运维复杂度和故障点。
3.  **国内最稳**：作为个人开发者，您肯定不希望折腾备案或者因为某个 JS 文件加载不出来导致登录页白屏。Casdoor 的国产基因保证了它在国内环境的绝对水土相服。
4.  **功能过剩优于功能不足**：Casdoor 的权限模型非常强大，未来如果您想做复杂的权限控制（比如“只有积分大于100的用户才能访问这个API”），Casdoor 开箱即支持，而 Logto 可能需要二次开发。

### 决策路径：
*   如果您的首要目标是 **“稳、省、全”** -> **Casdoor** (✅ 强烈推荐)
*   如果您的首要目标是 **“界面极其漂亮、不想看复杂配置”** 且 **“愿意额外部署 Postgres”** -> **Logto**

## 5. 附：Account 项目对接工作量预估 (以 Casdoor 为例)

1.  **后端**：
    *   移除 Supabase Auth 相关代码。
    *   引入 `casdoor-js-sdk` 或直接使用 OAuth2 标准库。
    *   实现 `/api/callback` 接口：接收 `code` -> 换取 `access_token` -> 解析用户信息。

2.  **前端**：
    *   将 `/login` 路由改为直接 `window.location.href = Casdoor登录页URL`。
    *   移除本地的登录/注册表单组件（全部托管给 Casdoor）。
    *   用户信息获取逻辑改为从后端 Session 或 ID Token 中解析。

这套方案实施后，您的 Account 项目将瘦身 60% 以上，只保留业务逻辑，认证逻辑全部解耦。

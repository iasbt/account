## 1. 引言 (Introduction)

### 1.1 编写目的
本文档旨在为 Account System（统一账户中心）的所有下游子系统提供一份详尽的接入指南与配置手册。
随着 Account System 架构升级至 V2.1（OIDC/OAuth 2.0 方案），原有的隐式流（Implicit Flow）已被废弃。所有新接入的子应用必须遵循本手册规范，采用 **Authorization Code Flow + PKCE** 模式，并使用 **RS256 非对称加密** 进行验证，以确保系统的安全性、稳定性与可维护性。

### 1.2 适用对象
*   **前端工程师**: 负责实现 OAuth 2.0 授权跳转、PKCE 挑战生成、Code 换 Token 及 Token 存储。
*   **后端工程师**: 负责对接 JWKS 端点进行 Token 签名验证与用户身份解析。
*   **运维工程师**: 负责配置环境变量、数据库白名单与网络策略。

### 1.3 核心价值
通过接入 Account System，子系统将获得以下核心能力：
*   **统一身份认证 (SSO)**: 用户只需登录一次，即可访问所有授权应用。
*   **安全性增强**: 采用业界标准的 OAuth 2.0 + PKCE + RS256 架构，杜绝令牌截获与篡改风险。
*   **集中式权限管理 (RBAC)**: 基于角色的访问控制，无需在每个应用中重复开发权限逻辑。
*   **用户画像共享**: 获取统一的用户基础信息（头像、昵称、邮箱等）。

## 2. 架构概览 (Architecture Overview)

### 2.1 系统拓扑
Account System 位于整个微服务架构的核心位置，作为 **Identity Provider (IdP)** 为各个 **Service Provider (SP)** 提供认证服务。

```mermaid
graph TD
    User[用户浏览器] -->|1. 访问应用| AppFrontend[子应用前端 (Gallery/Toolbox)]
    AppFrontend -->|2. 未登录 (生成 PKCE)| AccountSSO[Account SSO (Auth Center)]
    AccountSSO -->|3. 登录授权| AppCallback[子应用回调页]
    AppCallback -->|4. 携带 Code + Verifier| AccountAPI[Account API (Token Endpoint)]
    AccountAPI -->|5. 返回 Access + Refresh Token| AppFrontend
    AppFrontend -->|6. 携带 Token| AppBackend[子应用后端 API]
    AppBackend -->|7. 获取公钥 (JWKS)| AccountJWKS[Account JWKS Endpoint]
    AccountJWKS -->|8. 验证签名| AppBackend
```

### 2.2 关键组件
*   **Account Backend (IdP)**: 负责用户认证、Token 签发、应用注册管理及 JWKS 公钥分发。
*   **Sub-Application (SP)**: 消费 Token，利用 Token 中的 `sub` (Subject) 标识用户身份。
*   **PKCE (Proof Key for Code Exchange)**: 用于防止授权码被拦截的动态密钥交换机制。
*   **RS256**: 用于签名和验证 JWT 的非对称加密算法。

## 3. 接入准备 (Prerequisites)

在开始编码之前，必须完成以下准备工作。

### 3.1 应用注册 (Application Registration)
所有子应用必须在 Account 系统的数据库中注册，获取唯一的 `app_id`。
**注意**: V2.1 版本中，纯前端应用（Public Client）**不需要**也不应保存 `client_secret`。仅后端应用（Confidential Client）需要 Secret。

**操作步骤**:
1.  联系管理员访问 Account 数据库。
2.  在 `public.applications` 表中插入一条记录：
    ```sql
    INSERT INTO public.applications (app_id, name, secret, allowed_origins, is_active)
    VALUES (
      'gallery', -- 你的应用ID
      'Gallery App', -- 应用名称
      NULL, -- 前端应用不需要 Secret
      ARRAY['https://gallery.iasbt.com/auth/callback', 'http://localhost:5173/auth/callback'], -- 严格匹配的回调地址
      true -- 启用状态
    );
    ```

### 3.2 网络配置
确保子应用服务器能够访问 Account 系统的 API 接口。
*   **生产环境**: `https://account.iasbt.com` (公网访问)
*   **Docker 环境**: `http://account-backend:3000` (容器间通信)

## 4. 配置详解 (Configuration Guide)

为了规范化配置管理，建议所有子应用采用 `.env` 文件管理环境变量。

### 4.1 核心环境变量

| 变量名 | 说明 | 示例值 |
| :--- | :--- | :--- |
| `VITE_ACCOUNT_URL` | Account 统一地址 (前端/后端) | `https://account.iasbt.com` |
| `VITE_APP_ID` | 应用唯一标识，需与数据库一致 | `gallery` |
| `VITE_CALLBACK_URL` | 登录成功后的回跳地址 | `https://gallery.iasbt.com/auth/callback` |

### 4.2 配置加载示例 (Node.js/Vite)
```typescript
// config.ts
export const config = {
  accountUrl: import.meta.env.VITE_ACCOUNT_URL || 'https://account.iasbt.com',
  appId: import.meta.env.VITE_APP_ID || 'gallery',
  callbackUrl: import.meta.env.VITE_CALLBACK_URL || window.location.origin + '/auth/callback'
};
```

## 5. 集成流程 (Integration Workflow)

### 5.1 前端集成 (Frontend Integration)

前端主要负责引导用户登录并解析 Token。V2.1 必须实现 PKCE 流程。

#### 步骤 1: 生成 PKCE 并跳转
当检测到用户未登录时，生成 PKCE 挑战码并重定向：

```typescript
// pkce.ts (工具函数)
async function generateCodeChallenge(verifier: string) {
  const encoder = new TextEncoder();
  const data = encoder.encode(verifier);
  const digest = await window.crypto.subtle.digest('SHA-256', data);
  return btoa(String.fromCharCode(...new Uint8Array(digest)))
    .replace(/\+/g, '-')
    .replace(/\//g, '_')
    .replace(/=+$/, '');
}

// 登录逻辑
async function redirectToLogin() {
  const verifier = generateRandomString(64);
  const challenge = await generateCodeChallenge(verifier);
  
  // 存储 verifier 以便回调时使用
  sessionStorage.setItem('pkce_verifier', verifier);
  
  const params = new URLSearchParams({
    client_id: config.appId,
    redirect_uri: config.callbackUrl,
    response_type: 'code',
    code_challenge: challenge,
    code_challenge_method: 'S256',
    scope: 'openid profile email'
  });
  
  window.location.href = `${config.accountUrl}/api/oauth/authorize?${params.toString()}`;
}
```

#### 步骤 2: 处理回调 (Callback)
在回调页面（如 `/auth/callback`）使用 Authorization Code 换取 Token：

```typescript
// AuthCallback.tsx
useEffect(() => {
  const code = new URLSearchParams(window.location.search).get('code');
  const verifier = sessionStorage.getItem('pkce_verifier');
  
  if (code && verifier) {
    fetch(`${config.accountUrl}/api/oauth/token`, {
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify({
        grant_type: 'authorization_code',
        code,
        redirect_uri: config.callbackUrl,
        client_id: config.appId,
        code_verifier: verifier
      })
    })
    .then(res => res.json())
    .then(data => {
      localStorage.setItem('access_token', data.access_token);
      localStorage.setItem('refresh_token', data.refresh_token);
      window.location.href = '/';
    });
  }
}, []);
```

### 5.2 后端集成 (Backend Integration)

后端主要负责验证 Token 的合法性。V2.1 使用 JWKS 进行非对称验证。

#### 步骤 1: 安装依赖
```bash
npm install jsonwebtoken jwks-rsa
```

#### 步骤 2: 实现认证中间件 (Express)
```javascript
import jwt from 'jsonwebtoken';
import jwksClient from 'jwks-rsa';

const client = jwksClient({
  jwksUri: 'https://account.iasbt.com/.well-known/jwks.json'
});

function getKey(header, callback) {
  client.getSigningKey(header.kid, (err, key) => {
    callback(null, key?.getPublicKey());
  });
}

export const authMiddleware = (req, res, next) => {
  const token = req.headers.authorization?.split(' ')[1];
  
  jwt.verify(token, getKey, {
    algorithms: ['RS256'],
    issuer: 'https://account.iasbt.com',
    audience: 'gallery' // 验证接收方
  }, (err, decoded) => {
    if (err) return res.status(401).send('Invalid Token');
    req.user = decoded;
    next();
  });
};
```

## 6. 常见问题 (FAQ)

### Q1: 为什么提示 `redirect_uri_mismatch`？
*   **原因**: 传递的 `redirect_uri` 与数据库中注册的 `allowed_origins` 不完全一致（包括末尾斜杠）。
*   **解决**: 检查数据库配置，确保完全匹配。

### Q2: Token 验证失败 `invalid algorithm`？
*   **原因**: V2.1 强制使用 `RS256`，旧版代码可能还在使用 `HS256` 和 Secret。
*   **解决**: 升级验证逻辑，使用 `jwks-rsa` 获取公钥。

### Q3: Refresh Token 如何使用？
*   **流程**: 当 Access Token 过期（401）时，前端调用 `/api/oauth/token` (grant_type=refresh_token) 获取新 Token。
*   **注意**: 必须同时更新本地存储的 Refresh Token（轮换机制）。

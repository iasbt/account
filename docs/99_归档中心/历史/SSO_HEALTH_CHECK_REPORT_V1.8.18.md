# SSO Security Health Check Report (SSO 安全体检报告)

## 1. 核心安全风险与修复 (Critical Risks & Fixes)

### 1.1 回调地址伪造 (Redirect URI Spoofing) - ✅ FIXED
*   **风险**: 旧逻辑使用 `startsWith(origin)`，攻击者可利用 `https://example.com.evil.com` 绕过验证。
*   **修复**: 在 `ssoController.js` 中改为 `includes(redirect_uri)` 进行严格全等匹配。
*   **验证**: 已通过测试脚本验证，非白名单地址直接被拦截。

### 1.2 隐式流泄露 (Implicit Flow Leakage) - ✅ DEPRECATED
*   **风险**: `/sso/issue` 接口直接在 URL Fragment 中返回 Token，易被浏览器历史或 Referer 泄露。
*   **修复**: 标记该接口为 `410 Gone`，强制下线。
*   **替代**: 全面迁移至 OAuth 2.0 授权码模式。

### 1.3 授权码截获 (Authorization Code Interception) - ✅ IMPLEMENTED
*   **风险**: 在移动端或原生应用中，授权码可能被恶意软件截获。
*   **修复**: 引入 **PKCE (Proof Key for Code Exchange)** 机制。
    *   客户端生成 `code_verifier` 和 `code_challenge`。
    *   授权时提交 `code_challenge`。
    *   换票时提交 `code_verifier`，服务端进行 SHA-256 校验。
*   **状态**: 数据库迁移已完成 (`oauth_codes` 表新增字段)，逻辑已实装。

## 2. 架构设计缺陷与优化建议 (Architecture Defects & Optimization)

### 2.1 签名密钥误用 (Symmetric Key Misuse) - ⚠️ HIGH RISK
*   **现状**: 使用 `HS256` (对称加密)，所有验证方都需要共享同一个 Secret。
*   **风险**: 一旦任意一个应用（如 Gallery）泄露 Secret，即可伪造任意用户的 Token。
*   **方案**: 迁移至 **RS256 (非对称加密)**。
    *   Account 服务持有 **私钥 (Private Key)** 签名。
    *   应用服务持有 **公钥 (Public Key)** 验证 (通过 JWKS 端点自动获取)。
*   **对比**:
    *   *Keycloak/Casdoor*: 默认使用 RS256，并提供 `/.well-known/openid-configuration` 和 `/certs` 端点。

### 2.2 黑名单机制瓶颈 (Redis Blacklist Bottleneck) - ⚠️ PERFORMANCE
*   **现状**: JWT 本应无状态，但为了注销功能，目前将所有注销的 Token 存入 Redis 黑名单。
*   **风险**:
    *   Redis 宕机导致鉴权失效（或全部通过/全部拒绝）。
    *   每次请求都查 Redis，违背了 JWT 的无状态初衷。
*   **方案**: **短效 Access Token + 长效 Refresh Token**。
    *   Access Token 有效期设为 5-15 分钟 (无需撤销，过期自毁)。
    *   Refresh Token 存入数据库或 Redis (仅在刷新时校验，注销时撤销 Refresh Token)。
*   **对比**:
    *   *Keycloak*: Access Token 默认 5 分钟，SSO Session Idle 30 分钟。

# 产品选择报告：继续完善自建账户中心 vs 采用 Supabase OAuth Server

## Supabase 废弃声明（强制）
- Supabase 相关内容仅用于历史记录，当前已废弃且严禁使用。
- 现行方案唯一选择为“自建 Account + 标准 JWT + 自建 PostgreSQL（iasbt-postgres）”。
- 使用方式：`/api/auth/login`、`/api/auth/me`、`/api/sso/issue?target=...`，数据库结构以 `docs/支持文档/部署运维/附件/database.sql` 为准。

## 一、背景与需求重述

当前系统以 Account 项目为统一认证入口，承担用户登录、权限校验、应用接入管理与 SSO 回跳能力。核心目标包括：

- 让子应用快速接入 SSO，尽量做到“少改代码、少配参数”
- 回跳安全、稳定，具备可观测与审计能力
- 具备可持续的维护与演进能力，避免每个子应用重复造轮子

在此背景下，需要评估两种路线：

- 方案 A：继续完善当前 Account 项目（自建 SSO 中心）
- 方案 B：放弃当前 Account 为核心认证中心，转向使用 Supabase OAuth 2.1 Server 作为统一身份提供方

并以产品经理视角、用户视角与开发/运维视角进行全面分析，形成清晰、可执行的选择建议。

## 二、候选方案简述

### 方案 A：继续完善自建 Account

Account 作为独立前端项目，承载：

- 用户登录与会话初始化
- 应用接入管理（applications 表）
- 统一回跳策略与 allowlist 安全校验
- 访问审计与基础指标采集
- 子应用接入向导与最小代码片段输出

优势是对用户体验与业务流程可完全掌控，短期内可快速迭代；不足是需要持续维护认证协议与安全边界。

### 方案 B：使用 Supabase OAuth 2.1 Server 作为身份提供方

Supabase OAuth 2.1 Server 可以将当前 Supabase 项目作为 OAuth 2.1 / OIDC 身份提供方，为第三方应用提供授权能力（授权码 + PKCE、刷新、ID Token、JWKS 等）并与现有 Supabase Auth 体系集成。其优势在于标准化、可与第三方集成、减少安全风险与协议复杂度的自维护成本。Supabase 提供 OAuth 2.1/OIDC 能力、授权码与 PKCE、支持多端客户端、JWKS、公钥校验、无需共享 JWT Secret 等机制。来源说明见 Supabase 官方文档与功能页（参考：OAuth 2.1 Server 文档与功能说明）。[OAuth 2.1 Server 文档](https://supabase.com/docs/guides/auth/oauth-server) [Getting Started](https://supabase.com/docs/guides/auth/oauth-server/getting-started) [功能概览](https://supabase.com/features/oauth2-1-server)

## 三、多视角深度分析

### 1）产品经理视角

**业务可控性**
- 方案 A：可以高度定制登录体验、回跳规则、应用入口管理、审计与运营指标；更契合“统一入口 + 应用启动台”产品设想
- 方案 B：认证核心交给 Supabase，产品侧更偏“身份提供方”，但需额外自行建设应用启动台与运营功能

**成长性与生态**
- 方案 A：更像专用型认证中心，扩展到第三方生态时需要自行实现 OAuth/OIDC 标准接口
- 方案 B：天生符合 OAuth/OIDC 标准，适合后续做“开放平台/生态授权”，更容易和第三方系统集成

**产品路线的风险**
- 方案 A：自建逻辑多，容易积累“隐性协议债务”（回跳、授权、刷新、会话同步）
- 方案 B：协议和安全由 Supabase 维护，产品更聚焦在业务层，但需要适配 Supabase 的产品能力与迭代节奏

### 2）用户体验视角

**登录路径与一致性**
- 方案 A：当前 Account 可提供统一 UI，与现有业务一致
- 方案 B：OAuth Server 提供标准授权流程，但 UI 定制需额外配置“授权 UI 的承载页面”，路径更标准化

**稳定性与可预期性**
- 方案 A：稳定性依赖项目持续维护与测试覆盖
- 方案 B：授权流程由 Supabase 提供，稳定性更高，但仍需保证账户中心 UI 的稳定

**用户感知成本**
- 方案 A：统一入口，对用户来说更像“平台账号”
- 方案 B：如果做成“使用 Supabase 授权页面 + Account UI”，用户体验可以保持一致，但流程会更标准化

### 3）开发者体验与维护视角

**接入成本**
- 方案 A：已提供接入向导与最小片段，接入成本低，能做到“复制粘贴 + 简单配置”
- 方案 B：第三方需要按 OAuth 2.1 标准接入，接入文档严谨但门槛略高；若内部多为简单前端应用，需要额外封装“简化 SDK”

**安全边界与协议维护**
- 方案 A：需要持续维护允许回跳名单、回调逻辑、授权码交换逻辑、刷新机制、安全审计与异常处理
- 方案 B：OAuth/OIDC 的协议复杂性由 Supabase 承担，降低安全风险；官方实现支持 PKCE、JWKS 等标准能力，有利于安全与合规

**调试效率**
- 方案 A：前端可快速定位问题，但需要自行处理各种异常分支
- 方案 B：标准化授权流程更可观测，错误码与授权流程更一致，但出现问题需排查 Supabase 配置与 OAuth client

### 4）运维与成本视角

**运维压力**
- 方案 A：运维集中在前端与 Supabase Auth 的既有配置，仍需持续修复边界问题
- 方案 B：运维侧需要了解 OAuth Server 相关配置（OAuth client、redirect URI、授权 UI 等），但安全维护成本降低

**可审计性**
- 方案 A：可以定制审计，但需投入研发实现
- 方案 B：OAuth 流程天然有授权记录与标准结构，有助于后续合规与审计升级

**成本结构**
- 方案 A：研发人力成本高，但无需额外第三方成本
- 方案 B：依赖 Supabase 的产品能力与版本节奏，可能引入平台能力依赖，但节省研发维护成本

## 四、优缺点汇总（多维比较）

| 维度 | 方案 A：自建 Account | 方案 B：Supabase OAuth Server |
|---|---|---|
| 产品可控性 | 高，可塑性强 | 中，需适配平台能力 |
| 标准化程度 | 低到中 | 高（OAuth 2.1/OIDC） |
| 接入门槛 | 低 | 中（需 OAuth 规范） |
| 安全维护成本 | 高 | 低 |
| 扩展到外部生态 | 成本高 | 低，天然支持 |
| 迭代灵活性 | 高 | 受平台节奏影响 |
| 用户体验 | 可定制 | 标准化但可保持一致 |

## 五、风险分析

### 方案 A 风险
- 安全协议持续维护成本上升，出现“回跳未定义/异常缓存/会话同步漂移”
- 多项目接入时难保证一致性，技术债务累积
- 若未来对外开放生态，需要补齐 OAuth/OIDC 标准

### 方案 B 风险
- 依赖 Supabase 特性稳定性与收费策略
- 需要为子应用提供简化 SDK 或统一接入模板，否则接入门槛提升
- 现有 Account 的应用启动台、访问审计、运营能力需要额外保留或迁移

## 六、产品选择建议（推荐路线）

**推荐：不直接放弃 Account，而是升级为“业务入口 + 接入向导 + UI 中心”，同时逐步引入 Supabase OAuth Server 作为底层授权协议。**

理由：
- Account 现有价值集中在“统一入口、应用管理与向导”，这是 OAuth Server 不直接提供的
- Supabase OAuth Server 解决的是“身份提供与协议标准化”，可以与 Account UI 结合
- 这样可以得到：对外标准化（OAuth/OIDC） + 对内低门槛接入（向导/一键片段）

## 七、阶段性落地建议（可执行路线）

### 阶段 1：保持现有 Account 并完善“一键接入”
- 继续输出最小接入模板与 SDK 封装
- 扩展向导输出（已完成 Quick Start / Login URL / Button 片段）

### 阶段 2：引入 Supabase OAuth Server 作为协议层
- 在 Account 中保留统一登录 UI
- 子应用接入逐步迁移为 OAuth Client（授权码 + PKCE）
- 使用 Supabase 提供的 OAuth 2.1 Server 配置与客户端管理

### 阶段 3：统一审计与授权体系
- 基于 OAuth 授权记录 + Account 本地审计叠加
- 形成标准化授权治理与合规路径

## 八、面向三类角色的选择结论

### 产品经理结论
继续完善 Account 的产品能力是必须的；同时引入 Supabase OAuth Server 能减少长期协议负担并扩展生态能力。建议“双层架构”：Account 作为体验与运营层，Supabase OAuth 作为授权协议层。

### 用户结论
用户仍保持“单一账户入口”的一致体验，登录路径和回跳流程更加规范，稳定性与安全性更高，感知成本更低。

### 开发/运维结论
开发侧通过向导与模板维持低门槛接入；运维侧用标准 OAuth/OIDC 机制降低安全风险，减少长周期维护成本。

## 九、最终结论

**不建议直接放弃 Account。**  
建议采用“双层架构”的产品策略：Account 继续作为“统一入口 + 应用管理 + 接入向导”的体验层，同时把标准化授权协议逐步迁移到 Supabase OAuth Server。这样既保留当前产品价值，又获得标准 OAuth/OIDC 能力带来的生态扩展性与安全性优势。[OAuth 2.1 Server 文档](https://supabase.com/docs/guides/auth/oauth-server) [功能概览](https://supabase.com/features/oauth2-1-server)

## 十一、维护成本与演进影响

### 1）持续维护负担
- 自建方案需要持续维护回跳白名单、登录回调、会话同步、异常回退、跨域安全与缓存策略。
- OAuth Server 方案将协议复杂性外包给 Supabase，维护重点转为“OAuth client 注册与配置管理”。

### 2）团队能力要求
- 自建方案需要熟悉 OAuth/OIDC、PKCE、JWT、token 生命周期与安全策略的研发与安全能力。
- 采用 OAuth Server 后，团队更偏向“配置与管理”，安全协议复杂度由平台消化。

### 3）可运维性
- 方案 A 的可运维性取决于团队是否长期维护“SSO 协议与异常兼容性”。
- 方案 B 的可运维性更高，但依赖 Supabase 的可用性与版本稳定性。

## 十二、对当前产品与项目的影响分析

### 1）现有功能的去留
- 应用启动台、应用管理、接入向导：仍建议保留在 Account
- 授权协议：逐步改为 OAuth 2.1 / OIDC
- 访问审计：可保留现有 user_app_access，并增加 OAuth 授权审计维度

### 2）现有接入方式的兼容策略
- 保留现有 `redirect` 回跳模式作为“兼容层”
- 新增 OAuth 2.1 标准接入方式作为“升级路径”
- 向导内同时给出旧版片段与 OAuth 方式的“渐进式迁移方案”

---

## 十三、对开发者的最佳实践建议

1. **短期**：继续使用向导输出的 Login URL 与 AuthCallback 片段完成快速接入  
2. **中期**：逐步迁移为 OAuth 2.1 授权码模式，提升安全性  
3. **长期**：提供内部 SDK，对 OAuth 流程做薄封装，降低子应用接入门槛  

OAuth Server 标准能力说明（来源：Supabase 文档）  
- 支持 OAuth 2.1 / OIDC  
- 授权码 + PKCE  
- JWKS 用于第三方校验  
- 不需要共享 JWT Secret  
- 令牌与 RLS 策略一致  
[OAuth 2.1 Server 文档](https://supabase.com/docs/guides/auth/oauth-server) [Getting Started](https://supabase.com/docs/guides/auth/oauth-server/getting-started) [功能概览](https://supabase.com/features/oauth2-1-server)

## 十四、实施建议（里程碑）

### M1：继续完善 Account（1-2 个迭代）
- 稳定现有接入向导与最小代码片段
- 加强回跳错误提示与审计记录

### M2：引入 OAuth Server（1 个迭代）
- 开启 OAuth Server 功能
- 建立 OAuth client 管理流程与 UI
- 在 Account 内增加“OAuth 模式接入指南”

### M3：渐进迁移（持续）
- 逐步将核心子应用迁移到 OAuth 模式
- 收敛回跳与 token 交付方式

---

## 十五、最终建议摘要（给决策者的 30 秒结论）

- **不建议放弃 Account**：它承载了“统一入口 + 应用启动台 + 接入向导”的关键价值  
- **建议采用 Supabase OAuth Server 作为协议层**：提升安全性、标准化与生态扩展性  
- **最佳路径是融合方案**：Account 继续作为体验层，OAuth Server 作为标准授权层  

以上路径可以在不牺牲现有产品体验的前提下，最大化降低长期维护成本，并为未来开放生态打基础。

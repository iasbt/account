# 容器与项目功能全景手册

> 版本基线：`account@1.9.17`，Logto `1.37.1`  
> 适用对象：项目负责人、运维负责人、后端/前端协作成员  
> 目标：一次性讲清每个容器做什么、彼此如何通信、线上域名如何落地、未来如何安全演进。

---

## 1. 系统全景：你现在维护的是什么

你现在维护的不是单一应用，而是一个由“账户中心 + 外部认证 + 图库前端 + 运维面板”组成的组合系统。它有三个层次：

- **接入层**：域名、TLS、CDN/WAF、反向代理
- **应用层**：Account 后端、Account 前端网关、Gallery 前端、Logto
- **数据层**：Postgres、Redis

当前典型访问路径如下：

1) 用户访问 `iasbt.cloud`，进入 Gallery 前端；  
2) 用户需要登录时，跳转到 `account.iasbt.cloud` 或 Logto 认证流；  
3) 登录后回调到业务应用；  
4) 管理员通过 `logto-console.iasbt.cloud` 管理身份体系。

## 2. 容器清单与职责分工

### 2.1 account-frontend（Nginx 网关）

- 镜像来源：`correction-account-frontend`
- 端口：`80:80`
- 角色：
  - 统一入口网关
  - 承载 Account 前端静态资源
  - 代理 `/api` 到 `account-backend:3000`
  - 代理 Logto 两个子域到 `logto-core:3001/3002`
  - 代理根域 `iasbt.cloud` 到 `gallery-frontend:80`

这是你线上“流量分发中枢”。域名路由问题优先排查这个容器挂载的 `nginx.conf`。

### 2.2 account-backend（业务核心）

- 镜像来源：`correction-account-backend`
- 端口：`3000:3000`
- 角色：
  - Account 业务 API（登录、注册、管理、SSO、OIDC 等）
  - 与 Postgres 读写交互
  - 与 Redis 做缓存/黑名单/辅助状态
  - 提供 `/api/health`、`/metrics`

这是业务真核心。任何接口异常最终都要回到该容器日志与依赖连接状态。

### 2.3 gallery-frontend（图库前端）

- 镜像来源：`image-gallery-frontend`
- 端口：`5173:80`
- 角色：
  - 作为图库应用 UI 前端
  - 当前被根域 `iasbt.cloud` 指向为主入口

它是“业务入口页面”，通常不直接暴露复杂后端逻辑。

### 2.4 iasbt-postgres（主数据库）

- 镜像：`postgres:14-alpine`
- 端口：`5432:5432`
- 角色：
  - 主业务数据存储
  - 用户、应用、日志等核心表

建议尽快收敛为“内网访问优先”，减少公网暴露风险。

### 2.5 redis（缓存与辅助状态）

- 镜像：`redis:alpine`
- 端口：`6379:6379`
- 角色：
  - 缓存
  - 认证相关辅助状态（如黑名单等）

同样建议内网化，不建议长期开公网映射。

### 2.6 pgadmin（数据库可视化管理）

- 镜像：`dpage/pgadmin4`
- 端口：`8888:80`
- 角色：
  - DBA/管理员可视化运维工具

建议仅管理 IP 可访问，且启用强密码与最小权限账户。

### 2.7 portainer（容器运维控制台）

- 镜像：`6053537/portainer-ce`
- 端口：`9000:9000`
- 角色：
  - 看容器、日志、网络、卷、镜像
  - 运维操作入口

这是高敏感资产，原则上不能全网开放。

### 2.8 logto-core（认证中心）

- 镜像：`ghcr.io/logto-io/logto:latest`（你当前为 1.37.1）
- 端口：`3001/3002`（建议仅本机绑定，经 Nginx 子域转发）
- 角色：
  - OIDC 用户端（3001）
  - 管理控制台（3002）

它是“标准协议中心”，会对外暴露 OIDC 发现、JWKS、授权、token 相关端点。

### 2.9 logto-postgres（Logto 数据库）

- 镜像：`postgres:16-alpine`
- 端口：无公网映射
- 角色：
  - 仅服务 Logto

## 3. 域名与容器映射（运营口径）

当前推荐作为团队统一口径：

- `iasbt.cloud` -> `account-frontend(Nginx)` -> `gallery-frontend:80`
- `account.iasbt.cloud` -> `account-frontend(Nginx)` -> `account-backend:3000`（API）+ Account 前端静态
- `logto.iasbt.cloud` -> `account-frontend(Nginx)` -> `logto-core:3001`
- `logto-console.iasbt.cloud` -> `account-frontend(Nginx)` -> `logto-core:3002`

请注意：

- 域名决定“进哪个 server_name 路由块”。
- 端口决定“流量是否到达服务器/容器”。
- 容器网络决定“上游容器名能否被解析并连通”。

三者缺一不可。

## 4. 你问的关键问题：为什么访问 logto 根域会看到 JSON

现象：

- 无痕访问 `https://logto.iasbt.cloud/` 跳到 OIDC 发现文档，显示 JSON。

含义：

- 这表示你当前把该域名定位为“协议入口域名”，用于 OIDC 客户端自动发现授权地址、token 地址、jwks 地址。
- 这不是错误，而是“可观测的正常状态”。

为什么不是登录页：

- OIDC 的 `/oidc/auth` 需要最小参数（如 `client_id`、`redirect_uri`、`response_type` 等）。
- 直接裸访问授权端点会得到 `invalid_request`，这也是标准行为。

怎么使用才正确：

- 用户管理入口：`https://logto-console.iasbt.cloud/`
- 协议发现入口：`https://logto.iasbt.cloud/oidc/.well-known/openid-configuration`
- 业务应用应通过标准 OIDC 授权请求发起登录，而不是手工拼一个不完整 URL。

## 5. 端口与安全治理建议（按优先级）

### 5.1 P0：公网最小暴露

- 长期保留公网：`80/443`
- 管理端口白名单：`22/9000/8888`
- 数据与缓存端口内网化：`5432/6379`
- 评估收敛 `3000` 直出，仅走 Nginx 反代

### 5.2 P1：边界与应用双层防护

- EdgeOne：WAF、限速、异常流量策略
- Nginx：严格 `server_name`、限制非法 Host
- 应用层：限流、中间件审计、敏感操作日志

### 5.3 P1：证书与域名卫生

- 子域名新增后立即补证书
- 下线旧域名后同步清理配置引用
- 证书覆盖策略以 `iasbt.cloud` 体系为主

### 5.4 P2：运维流程标准化

- 每次发布固定执行冒烟 URL 集
- 每周导出端口清单与防火墙比对
- 每月一次域名证书覆盖审计

## 6. 变更流程建议（你可以直接执行的模板）

1. 本地改 `deploy/correction/nginx.conf` 与脚本变量。  
2. 运行 `npm run lint && npm run typecheck`。  
3. 执行 `.\deploy_remote.ps1 "msg"`。  
4. 验证四个核心域名 URL（root、health、oidc、console）。  
5. 在 Portainer 看容器日志与重启次数。  
6. 在 EdgeOne 看证书和回源状态。

## 7. Logto 1.37.1 后续是否要跟官方升级

建议：**要跟，但不要“自动追最新版”**，采用“评估后升级”。

原因：

- Logto 属于认证核心组件，升级可能包含安全修复与协议兼容优化。
- 但它也可能带来行为变更，尤其在会话、CSP、回调策略方面。

推荐策略：

- 使用“固定版本标签”而不是长期 `latest` 漂移。
- 先在测试环境验证：登录、回调、刷新 token、登出、JWKS、管理员后台。
- 通过冒烟后再生产发布。
- 保留回滚点（镜像与配置快照）。

## 8. 故障定位速查表

### 问题 A：域名可访问但页面不对

- 查 EdgeOne 回源域名与主机头
- 查 `nginx.conf` 的 `server_name` 命中
- 查 `location /` 是否被意外覆盖

### 问题 B：反代 502/504

- 查上游容器是否存活
- 查容器网络是否互通
- 查上游主机名是否可解析

### 问题 C：Logto 报 invalid_request

- 判断是不是手工裸访问协议端点
- 补齐标准授权参数，按应用流程发起

### 问题 D：证书主机名错误

- 查证书 SAN 是否覆盖该域名
- 查访问域名是否误用了旧别名
- 查 EdgeOne 证书绑定和生效状态

---

## 9. 推荐的长期架构收敛方向

- 入口统一：所有业务域名先入 Edge，再入 `account-frontend` 网关。
- 配置统一：域名只在少数变量中定义（`PRIMARY_DOMAIN`、`ACCOUNT_PUBLIC_URL`、`LOGTO_*`）。
- 文档统一：对外说明统一用 `.cloud` 域名，避免 `.com` 历史残留。
- 安全统一：管理端口白名单化、数据库端口内网化、审计日志可追溯化。

---

## 10. 一句话总结

你当前系统已经具备“可持续运营”的骨架：  
**根域走 Gallery、账户走 account 子域、认证走 logto 子域、协议与管理分离**。  
下一步重点不是再堆功能，而是做“端口最小暴露 + 版本可控升级 + 证书与文档统一口径”的安全收口。

# 域名、端口、容器映射与防火墙安全指南

> 适用环境：`iasbt.cloud` 生产环境（EdgeOne + Docker + Nginx）  
> 目标：让你快速看清“哪个域名落到哪个端口、哪个容器、哪条网络路径”，并能基于该映射做精准防火墙策略。

## 1. 先建立一个核心认知

在你当前架构中，**域名与端口不是一一直接绑定**，而是经过一层反向代理：

`浏览器 -> EdgeOne(443/80) -> 服务器 80 -> account-frontend(Nginx) -> 目标容器端口`

这意味着：

- 对公网来说，主要开放的是 `80/443`。
- 具体服务（Account API、Logto 用户端、Logto Console、Gallery 前端）是由 Nginx `server_name` 分流。
- Docker 容器内部端口（3000/3001/3002/80）通常不应直接暴露公网访问。

## 2. 你现在的关键域名映射

基于当前配置，推荐按下列口径理解：

- `https://iasbt.cloud` -> `account-frontend` 中 Nginx -> `gallery-frontend:80`
- `https://account.iasbt.cloud` -> `account-frontend` 中 Nginx -> 静态页 + `/api` 转 `account-backend:3000`
- `https://logto.iasbt.cloud` -> `account-frontend` 中 Nginx -> `logto-core:3001`
- `https://logto-console.iasbt.cloud` -> `account-frontend` 中 Nginx -> `logto-core:3002`

说明：

- `logto.iasbt.cloud` 根路径返回/跳转 OIDC 发现文档是正常行为，代表该域名承担 OIDC 协议入口职责。
- 看到 `oidc.invalid_request`（例如缺少 `client_id`）通常是“手工访问了协议端点”，不是服务故障。

## 3. 你可以在哪看“可视化界面”

### 3.1 Portainer（最直观）

用途：

- 看容器运行状态、端口映射、所属堆栈。
- 看容器网络连接（哪个 network）。
- 看容器日志与重启记录。

建议检查项：

- `account-frontend` 是否 `80:80`。
- `account-backend` 是否 `3000:3000`（是否保留公网映射可按安全策略决定）。
- `logto-core` 是否只绑定 `127.0.0.1:3001/3002`（当前是更安全做法）。
- `gallery-frontend` 是否与 `account-frontend` 处在可互通网络（便于 Nginx 反代）。

### 3.2 EdgeOne 控制台（公网入口）

用途：

- 看域名证书覆盖情况（SNI/CN/SAN）。
- 看 DNS 与回源配置。
- 看 WAF、速率限制、地理封禁、Bot 策略。

建议检查项：

- `iasbt.cloud`、`account.iasbt.cloud`、`logto.iasbt.cloud`、`logto-console.iasbt.cloud` 均有正确证书。
- 回源协议与源站监听一致（通常 Edge HTTPS 回源到源站 HTTP:80）。

## 4. 本地文件怎么看映射关系

你项目中最关键的三个文件：

1. `deploy/correction/nginx.conf`  
   决定**域名到上游容器**的分流规则（`server_name` + `proxy_pass`）。

2. `deploy/correction/docker-compose.yml`  
   决定容器的端口暴露、网络、挂载。

3. `deploy_remote.ps1`  
   决定部署时如何写入 `.env`、如何重启容器、如何做健康检查。

排障顺序建议：

1) 先看域名是否命中正确 `server_name`；  
2) 再看 `proxy_pass` 上游容器名是否可解析；  
3) 再看容器网络是否连通；  
4) 最后看防火墙是否误拦截。

## 5. 防火墙“精准开关端口”建议

### 5.1 建议公网开放

- `80/tcp`：HTTP 回源与跳转用途
- `443/tcp`：HTTPS 终端入口（若由 EdgeOne 托管，可按你的回源模型保留最小暴露）

### 5.2 建议限制为内网/白名单

- `22/tcp`：仅你的办公出口 IP
- `9000/tcp`（Portainer）：仅管理 IP，绝不全网开放
- `8888/tcp`（pgAdmin）：仅管理 IP 或关闭公网
- `5432/tcp`（Postgres）：强烈建议关闭公网，仅容器网络
- `6379/tcp`（Redis）：强烈建议关闭公网，仅容器网络
- `3000/tcp`（Backend）：建议只给内网或直接取消公网映射

### 5.3 可执行安全加固动作

- 在云防火墙按“源 IP + 端口”双维度放行。
- 为管理端口启用分层鉴权（如 VPN/堡垒机 + 强密码 + MFA）。
- 保留 `80/443` 对外，其他端口默认拒绝。
- 每次发布后执行一次端口基线核查（防止临时调试端口忘记关闭）。

## 6. 如何验证“域名 -> 端口 -> 容器”是否生效

建议用以下思路做冒烟验证（PowerShell 中使用 `curl.exe`）：

- `curl.exe -I https://iasbt.cloud`
- `curl.exe -I https://account.iasbt.cloud/api/health`
- `curl.exe -I https://logto.iasbt.cloud/oidc/.well-known/openid-configuration`
- `curl.exe -I https://logto-console.iasbt.cloud/console`

判定标准：

- 返回码符合预期（200/302）。
- Location 跳转域名与目标一致。
- 证书主机名与访问域名一致。

## 7. 常见误区

- 误区 1：看到容器端口映射就认为必须公网开这个端口。  
  实际上很多端口可以只用于容器网络，不必暴露公网。

- 误区 2：把“域名映射”与“防火墙端口”做一一硬绑定。  
  实际上域名分流在 Nginx/Edge 层完成，防火墙只控制传输层入口。

- 误区 3：把 OIDC 协议端点当网页入口。  
  OIDC 端点返回 JSON 或 invalid_request 很常见，不等于故障。

## 8. 建议你长期保持的运维节奏

- 每次变更域名路由后：检查 `nginx.conf` + 线上 `curl.exe -I` 四条核心 URL。
- 每次发布后：核查防火墙规则是否出现“临时放开未回收”。
- 每周一次：在 Portainer 导出容器端口列表，和基线清单比对。
- 每月一次：审计证书覆盖域名，避免新增子域名后漏签发。

## 1. 系统概述与核心理念

### 1.1 什么是 IASBT 账号中心？

IASBT 账号中心是整个系统生态的“海关”与“身份证发放处”。无论是内部的管理后台，还是外部的其他子应用，所有的用户都必须在这里进行注册、登录、修改密码等操作。

它的核心使命有两个：

1. **身份认证 (Authentication)：** 证明“你是谁”。（比如通过比对账号和密码）。
    
2. **单点登录 (SSO - Single Sign-On)：** 提供“一次登录，处处通行”的体验。就像去游乐园，你在大门口买了一张通票（Token），之后去玩过山车、碰碰车（各个子应用），只需要出示这张通票，而不需要重新买票。
    

### 1.2 设计哲学

在代码的世界里，很容易陷入过度设计的泥潭。我们的账号模块应当保持克制，接口设计要直白，职责划分要明确。不堆砌无用的逻辑，让每一行代码都有其存在的意义，用最少的代码实现最坚固的安全防线。

## 2. 技术栈与架构解析

不要被各种英文缩写吓倒，我们可以把这套系统想象成一家运转良好的餐厅。

### 2.1 Node.js 与 Express (餐厅的引擎与服务员)

- **Node.js** 是运行环境，它让 JavaScript 这种原本只能在浏览器里运行的语言，可以在服务器上运行。它是整个餐厅的水电系统。
    
- **Express** 是基于 Node.js 的框架。它就像是餐厅里的“服务员”（Controller 层）。当用户（顾客）发来一个网络请求（点单），Express 负责接收这个请求，把它交给后面的厨房（Service 层）处理，然后再把结果端回给用户。
    

### 2.2 PostgreSQL (餐厅的账本 / 保险柜)

PostgreSQL（简称 PG）是一个强大的关系型数据库。

- 它负责永久性地存储数据，比如用户的邮箱、加密后的密码、注册时间、以及系统中注册了哪些第三方应用。
    
- **关系型**的意思是，数据像 Excel 表格一样规规矩矩地存放在行和列中，严谨且不易出错。
    

### 2.3 Redis (餐厅的便签纸 / 临时储物柜)

在你的代码中，`utils/verificationStore.js` 使用了 `ioredis`。

- Redis 是一个**内存数据库**。它的读写速度极快，但存放在里面的东西断电即焚（当然可以配置持久化，但通常用于缓存）。
    
- 在我们的系统里，它就像一张“便签纸”。当用户请求验证码时，我们把“邮箱 -> 123456”写在 Redis 里，并设定 5 分钟后自动销毁这张便签。这比把验证码写进厚重的 PostgreSQL 账本里要高效得多。
    

### 2.4 JWT (游乐园的腕带通票)

JWT (JSON Web Token) 是目前最流行的跨域认证解决方案。

- 当用户密码核对正确后，我们不发实体门票，而是给用户发一条加密的字符串（Token）。
    
- 这个字符串里包含了用户的 ID、名字等信息，并且盖了我们服务器独有的“防伪公章”（`ssoSecret`）。只要公章是真的，我们就认这张票。

## 3. 核心功能模块深度拆解

我们将结合你现有的代码结构，从外到内（从 Controller 到 Service）拆解业务逻辑。

### 3.1 验证码与注册模块

**业务场景：** 用户输入邮箱，点击发送验证码。

1. **服务员接单 (`authController.js - sendVerificationCode`)**： 服务员拿到了顾客的邮箱 `email`，告诉后厨需要发送验证码。
    
2. **后厨处理 (`authService.js - sendVerificationCode`)**：
    
    - 生成一个 6 位的随机数。
        
    - 调用 `setVerificationCode` 把这个数字贴到 Redis 这张便签纸上，并设定 300 秒（5分钟）倒计时。
        
    - 调用 `sendEmail`，把验证码包装成漂亮的 HTML 邮件发给用户。
        
3. **顾客注册 (`authController.js - register`)**：
    
    - 顾客拿着收到的验证码、名字、密码来注册。
        
    - **第一步：防伪校验**。后厨去 Redis 里找这张便签，如果找不到（过期了）或者数字对不上，直接打回。
        
    - **第二步：查重**。去 PostgreSQL 账本里查一下，这个邮箱或者名字是不是已经存在了。
        
    - **第三步：不可逆加密**。这是非常关键的一步！代码中使用了 `bcryptjs`。密码绝对不能明文存进数据库（比如直接存 "123456"）。`bcryptjs` 就像一台绞肉机，把 "123456" 绞碎成一串毫无规律的乱码（哈希值）。这是一条单行道，哪怕是数据库管理员，也没办法把这串乱码还原成原来的密码。
        
    - **第四步：发证**。在数据库里记录下这个新用户，并生成一张 JWT 门票给用户。销毁 Redis 里的验证码便签。
        

### 3.2 登录模块 (`authController.js - login`)

1. 用户提交账号和密码。
    
2. 系统去 PostgreSQL 中找到这个用户的一行数据。
    
3. 取出数据库里存放的“绞碎的肉”（哈希密码），再把用户刚输入的密码也扔进同样的绞肉机绞一次。如果两次绞出来的结果一模一样（`bcryptjs.compare`），说明密码正确。
    
4. 调用 `generateToken`，用我们的 `ssoSecret` 把用户的 ID、角色信息打包成一个 JWT 返回给前端。前端把这个 Token 存起来，以后每次请求都带上它。
    

### 3.3 个人资料与密码修改

这部分逻辑相对清晰：

- **修改密码**：必须要求用户提供旧密码，验证通过后，再用绞肉机处理新密码并更新到数据库。
    
- **修改资料**：接收新的头像或昵称，拼装 SQL 语句（`UPDATE public.users SET ...`），执行更新。
    

### 3.4 退出登录 (`authController.js - logout`)

系统现有的代码中，退出登录的逻辑预留了处理 SSO 重定向的空间。

- 前端发起 Logout 请求。
    
- 如果请求带有 `target` 参数，并且通过了 `isValidRedirectTarget` 的白名单校验，系统会让前端跳转回指定的应用页面。这是为了防止**开放重定向漏洞**（黑客伪造一个退出链接，把用户引导到钓鱼网站）。

## 4. 数据库与缓存设计规范

良好的数据结构是系统稳定运行的基石。

### 4.1 用户表 (`public.users`)

这是系统的核心表。

- **id**: 必须使用 UUID（全局唯一标识符），而不是 1,2,3 这种递增的数字。递增数字容易被黑客猜出系统有多少用户，而 UUID 是一长串随机字符，极其安全。
    
- **name / email**: 作为登录凭证，必须在数据库层面设置 `UNIQUE`（唯一约束）。
    
- **password**: 长度必须留够，因为 bcrypt 加密后的字符串通常在 60 个字符左右。
    
- **is_admin**: 布尔值，用于区分普通用户和超级管理员。
    
- **created_at / updated_at**: 审计字段，记录账号的生命周期。
    

### 4.2 应用注册表 (`public.applications`)

在你的 `ssoController.js` 中提到了这个表。这是单点登录的枢纽。

- 你要接入一个新系统（比如一个相册应用 Gallery），必须先在这里登记。
    
- **app_id**: 子应用的唯一标识。
    
- **allowed_origins**: 一个数组（比如 `['https://gallery.example.com']`）。只有这些白名单里的网址来申请 SSO 登录，我们的系统才会放行，防止恶意网站来“骗取”门票。
    

### 4.3 Redis 键值设计

目前使用了 `verify:{email}` 的格式存验证码。在未来的扩展中，如果我们要管理 Token 黑名单，可以设计 `blacklist:token:{token_id}` 的键值结构。

## 5. 安全规范与风险防御 (护城河的建立)

对于一个初学者来说，理解代码怎么写是第一步，理解代码为什么会**被黑客攻破**是进阶的必经之路。身份认证系统是最容易被攻击的靶子。

### 5.1 伪随机数漏洞 (重要修正)

在当前的 `authService.js` 中：

JavaScript

```
const code = Math.floor(100000 + Math.random() * 900000).toString();
```

**风险：** `Math.random()` 在计算机科学中被称为“伪随机”。它是由一个内部算法算出来的，如果黑客连续疯狂请求 1000 次验证码，收集到足够多的样本，他就能用数学公式算出你系统下一个生成的验证码是多少。这意味着他可以任意重置别人的密码！ **防线：** 必须使用 Node.js 提供的密码学安全随机模块。

JavaScript

```
import { randomInt } from "crypto";
const code = randomInt(100000, 1000000).toString(); // 这是真正无法预测的随机数
```

### 5.2 暴力破解与限流 (Rate Limiting)

**风险：** 黑客写一个脚本，一秒钟试 1000 次密码，或者一秒钟发 1000 次验证码短信，直接把服务器搞崩溃，或者把你的邮件服务账单刷爆。 **防线：**

- 在 Express 引入 `express-rate-limit` 中间件。
    
- 规定：同一个 IP，针对登录接口，15 分钟内最多尝试 5 次；针对发验证码接口，1 分钟只能请求 1 次。
    

### 5.3 Token 泄漏与无状态的诅咒

**风险：** JWT 就像发出去的纸质门票，上面写了“7天内有效”。如果这张票被小偷（黑客）偷走了，在接下来的 7 天里，小偷都可以畅通无阻。你的 `logout` 接口仅仅返回了“已安全退出”，但这张门票本身并没有作废。 **防线：** 我们需要打破纯粹的“无状态”，引入“黑名单”机制。 当用户点击退出登录时，我们把这张门票提取出来，扔进 Redis 的黑名单里，设置过期时间为门票的剩余寿命。每次系统收到门票时，都要先去 Redis 看一眼：这张票是不是在黑名单里？如果在，直接拒绝。

## 6. 单点登录 (SSO) 的前世今生与标准演进

我们来看看你代码里的 `ssoController.js`。这是整个系统最复杂，也是最需要重构的地方。

### 6.1 当前代码的模式：隐式授权流 (Implicit Flow)

你的代码目前是这样工作的：

1. 子应用带着 `target` 跳转到账号中心。
    
2. 账号中心验证你是合法的应用，生成 `accessToken`。
    
3. 把 Token 直接拼在网址的尾巴上：`url.hash = access_token=xxx...`，然后重定向回子应用。
    

**风险：** 这在业界叫做“隐式授权流”。Token 直接暴露在了前端的网址上！任何人只要站在用户背后看一眼屏幕，或者查看浏览器的历史记录，就能直接把 Token 抄走。这种做法在现代安全标准中已经被**明确废弃**。

### 6.2 进阶演进：授权码模式 (Authorization Code Flow)

作为未来的重构目标，你应该将其升级为标准的 OAuth 2.0 授权码模式。这看起来复杂，但逻辑非常严密：

1. **发一次性凭证：** 子应用跳转过来，用户登录成功后，账号中心**不给 Token**，而是给一个极短寿命（比如 1 分钟有效）的“授权码 (Code)”，拼在 URL 后重定向回去（`?code=abc123`）。
    
2. **后端秘密交易：** 子应用的前端拿到 Code，发给自己的**子应用后端**。
    
3. **换取真门票：** 子应用后端带着这个 Code，以及它自己的 `App Secret`（应用密码），在服务器底层悄悄向账号中心发起请求。
    
4. **验证发票：** 账号中心一看，Code 是真的，App Secret 也是对的，于是把真正的 `accessToken` 发给子应用的后端。
    

**为什么这样更安全？** 整个过程中，真正的 Token 永远只在“服务器与服务器”之间传输，黑客即便攻破了用户的浏览器，也拿不到 Token。这才是业界开源软件（如 Keycloak、Casdoor）的标准做法。

## 7. 开发、测试与部署规范

要让代码从你个人的电脑，平稳地运行到云服务器上，需要遵循严格的工程化规范。

### 7.1 环境变量 (.env) 的管理

绝对不要把密码、Secret 密钥硬编码在代码里！ 你的项目中有一个 `.env.example`。在本地开发时，你需要复制一份重命名为 `.env`。 这里面应该包含：

- 数据库连接地址 (`DB_HOST`, `DB_USER`, `DB_PASS`)
    
- Redis 连接地址 (`REDIS_URL`)
    
- Token 签名的密钥 (`SSO_SECRET`) —— 这个字符串必须长且随机。
    
- 发邮件的 SMTP 配置信息。
    

### 7.2 日志记录 (Logging)

代码里的 `console.error` 只能在黑框框里看看。系统上线后，你需要引入专门的日志库（如 `winston` 或 `pino`）。

- 不仅要记录报错信息（Error），还要记录关键的操作轨迹（Info：比如谁在什么时间登录了系统）。
    
- 日志是排查线上 Bug 和追踪黑客攻击的唯一线索。
    

### 7.3 安全层面的中间件配置

在你的 `app.js` 或主入口中，必须配置 `helmet` 中间件。它可以自动帮你设置各种安全的 HTTP 响应头，比如防御 XSS（跨站脚本攻击）、禁止别人用 iframe 嵌套你的网站（点击劫持防御）等。

## 8. 项目未来演进路线图

作为一名刚刚入门并准备大展拳脚的开发者，你的系统目前已经具备了良好的骨架。为了让它成为一个企业级的 SSO 服务，你可以将以下内容列为未来的迭代计划：

### 第一阶段：安全与体验修复 (短期)

1. **修复随机数问题：** 将所有的 `Math.random` 替换为 `crypto.randomInt`。
    
2. **补齐 Token 黑名单：** 完善 Logout 逻辑，利用 Redis 实现 Token 的主动失效。
    
3. **防暴破限制：** 引入限流机制，保护短信/邮件接口不被恶意刷爆。
    

### 第二阶段：协议标准化 (中期)

1. **废弃隐式流：** 重写 `ssoController.js`，引入标准的授权码模式。你可以尝试引入 `oauth2-server` 这样的成熟类库来帮你处理复杂的协议状态。
    
2. **引入非对称加密 (RS256)：** 目前你的 Token 是用对称加密（所有系统共用一个密码）。未来应该让账号中心持有**私钥**来签发 Token，对外公开一个接口提供**公钥**。子系统拿着公钥去验证 Token 真伪，实现解耦。
    

### 第三阶段：功能丰富与可观测性 (长期)

1. **多因素认证 (MFA)：** 除了密码，再加入动态口令（类似于谷歌验证器），进一步提升安全性。
    
2. **会话管理仪表盘：** 在管理员后台，能够看到当前有哪些用户在线，分别登录了哪些子系统，并允许管理员一键“踢人下线”。

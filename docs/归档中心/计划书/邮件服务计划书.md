## 1. 项目背景与现状痛点

当前 Account 系统的邮件模块仅具备最基础的“开发者视角”功能（Raw HTML 编辑 + 简单测试），存在以下严重痛点，无法满足“行业惯例”的运营需求：
1.  **交互原始**: 仅提供代码编辑器，非技术人员无法维护模板，缺乏所见即所得（WYSIWYG）能力。
2.  **配置硬编码**: SMTP 服务配置依赖环境变量，无法在后台动态切换或轮换通道。
3.  **黑盒运行**: 缺乏发送记录（Logs），无法追溯邮件是否送达、何时送达、失败原因。
4.  **同步阻塞**: 发送逻辑与 HTTP 请求耦合，一旦 SMTP 响应慢，直接导致用户接口超时。
5.  **缺乏监控**: 没有发送量、成功率等关键指标的可视化仪表盘。

本计划书旨在重新定义“邮件服务”模块，将其从一个简单的工具函数升级为**全功能的邮件发送与管理平台**。

## 2. 核心功能模块设计 (Functional Architecture)

根据行业标准（对标 SendGrid, Postmark, Auth0 Email Provider），新版邮件服务将包含以下四大核心子模块：

### 2.1 仪表盘 (Dashboard & Monitoring)
**目标**: 一眼看清服务健康度。
*   **实时指标**: 今日发送量、本周累计、当前队列堆积数。
*   **趋势图表**: 使用折线图展示 24 小时内的发送成功/失败趋势。
*   **服务状态**: 实时检测 SMTP 连接状态、Redis 队列健康度。
*   **用量预警**: 当失败率超过阈值（如 5%）时在 UI 高亮报警。

### 2.2 服务配置中心 (Provider Configuration)
**目标**: 脱离代码/环境变量的动态配置。
*   **可视化表单**: 在后台直接配置 SMTP Host, Port, Secure, Auth User/Pass。
*   **多通道支持**: 预留架构支持切换不同服务商（如 AWS SES, SendGrid API, Custom SMTP）。
*   **发件人管理**: 配置默认的 "From Name" 和 "Reply-To" 地址。
*   **连接测试**: 配置保存前的实时连通性验证（Ping/Auth Check）。

### 2.3 智能模板引擎 (Smart Template Engine)
**目标**: 运营人员可用的模板管理。
*   **双模式编辑器**:
    *   **高级模式**: 现有的 Monaco Code Editor (保留给开发者)。
    *   **设计模式**: 富文本编辑器 (Rich Text) 或 拖拽式构建器，支持图片上传与样式调整。
*   **动态变量注入**: 侧边栏列出可用变量（如 `{{code}}`, `{{username}}`, `{{link}}`），点击即可插入。
*   **多设备预览**: 增加 手机/平板/桌面 的响应式预览切换。
*   **版本控制**: 记录模板的修改历史，支持一键回滚到旧版本。
*   **默认模板库**: 内置行业标准的精美 HTML 邮件模板（注册、重置、通知、营销）。

### 2.4 审计日志与追踪 (Audit Logs & Tracking)
**目标**: 每一封邮件都有迹可循。
*   **全量记录**: 记录每一封邮件的 `MessageID`、接收者、模板类型、发送时间、耗时。
*   **状态追踪**: `Queued` (入队) -> `Sending` (发送中) -> `Sent` (成功) / `Failed` (失败)。
*   **失败诊断**: 详细记录 SMTP 返回的错误码与错误信息（如 "550 User not found"）。
*   **一键重发**: 针对失败的邮件，提供“手动重试”按钮。

## 3. 技术架构升级 (Technical Architecture)

为了支撑上述功能，后端架构需进行深层次重构。

### 3.1 数据库设计 (Schema Update)
新增以下核心表结构：

```sql
-- 1. 邮件服务配置表 (动态配置)
CREATE TABLE email_providers (
    id SERIAL PRIMARY KEY,
    name VARCHAR(50) NOT NULL, -- e.g., "System Default"
    host VARCHAR(255) NOT NULL,
    port INTEGER NOT NULL,
    secure BOOLEAN DEFAULT TRUE,
    auth_user VARCHAR(255),
    auth_pass_encrypted TEXT, -- 加密存储
    is_active BOOLEAN DEFAULT FALSE,
    created_at TIMESTAMP DEFAULT NOW()
);

-- 2. 邮件发送日志表 (审计核心)
CREATE TABLE email_logs (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    recipient VARCHAR(255) NOT NULL,
    template_type VARCHAR(50) NOT NULL,
    subject VARCHAR(255),
    status VARCHAR(20) NOT NULL, -- 'queued', 'sent', 'failed'
    error_message TEXT,
    retry_count INTEGER DEFAULT 0,
    sent_at TIMESTAMP,
    created_at TIMESTAMP DEFAULT NOW()
);
```

### 3.2 异步队列机制 (Async Queue)
*   **引入 Redis + BullMQ**:
    *   将邮件发送从同步操作（`await sendEmail`）转变为生产者-消费者模型。
    *   **API 响应**: 接口（如注册）只需将任务推入 Redis 即可立即响应前端，极大提升接口速度。
    *   **Worker**: 独立的后台 Worker 进程负责从 Redis 消费任务并调用 SMTP 发送。
    *   **重试策略**: 配置指数退避策略（Exponential Backoff），网络抖动自动重试。

## 5. 预期成果与价值

完成本次重构后，Account 系统的邮件服务将达到以下标准：

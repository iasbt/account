# SSO 与 OAuth Server 融合路线计划书（≥4000字）

## 0. 前言与目标摘要

本计划书基于既有 Account 统一账号中心，遵循“体验层与协议层分离”的路线：保留 Account 作为统一入口、应用启动台与接入向导，同时引入 Supabase OAuth Server 作为标准化授权协议层。目标是在不牺牲现有产品体验的前提下，实现对外部第三方与内部多子应用的标准化授权能力，并降低长期安全协议维护成本。计划兼顾产品经理视角、用户体验、研发交付与运维治理，形成可落地的阶段性实施路线与验收机制。

关键要求与约束：
- 保持 Account 站点的统一登录体验与应用管理能力
- OAuth Server 配置需符合 Supabase 配置要求，包括站点 URL 与授权路径（Authorization Path /oauth/consent）
- 子应用接入尽量“懒人化”，仅需复制配置与最小代码片段
- 兼容当前回跳机制（redirect + allowlist）并提供渐进迁移路径
- 明确里程碑与可检验指标，避免“上线即变成技术债”

## 1. 业务需求重述与问题根因分析

### 1.1 业务需求重述

1. **统一身份入口**：用户通过 Account 进行登录、注册、找回密码等操作，并以此进入各子应用。
2. **跨应用单点登录**：用户在任何子应用或 Account 登录后，其他子应用能共享认证状态，避免重复登录。
3. **安全回跳与一致体验**：登录后优先返回来源应用，回跳路径必须可控、安全、可审计。
4. **低门槛接入**：子应用只需最小化配置，即可接入账号中心。
5. **可观测与可治理**：需要有可追踪的授权记录、访问审计、回跳失败率与异常统计。
6. **长期可扩展**：未来可能开放第三方集成或生态接入，需要遵循标准协议（OAuth/OIDC）。

### 1.2 当前痛点与根因

1. **协议复杂度逐渐上升**  
现有方案侧重“快速回跳 + token 交付”，缺少 OAuth/OIDC 标准的完整链路，随着子应用增多和外部合作加入，容易出现“非标准流程 + 非标准错误处理 + 不可复制的接入方式”的技术债。

2. **安全边界维护成本高**  
回跳白名单、token 交付、session 同步、刷新策略等都需要项目内部自行处理，并随浏览器策略变化、第三方应用环境变化不断调整。

3. **扩展到外部生态缺乏通用接口**  
内部应用可以用自定义回跳机制，但对外合作方通常需要 OAuth/OIDC，缺乏标准授权接口会成为阻力。

4. **运维与排障成本偏高**  
异常回跳、缓存问题、会话失效与授权失败均需要自建排查方式。缺乏标准化授权日志结构，导致“排查靠经验”。

### 1.3 结论

Account 作为体验层价值很高，但协议层需要更标准化、可扩展、可治理的机制。引入 Supabase OAuth Server 是系统升级的最佳方向，同时保留现有“快速接入”体验。

## 2. 目标架构与设计原则

### 2.1 总体架构

目标是形成“双层架构”：

- **体验层（Account）**  
  负责统一登录 UI、应用管理、接入向导、回跳兼容与可视化管理。
- **协议层（Supabase OAuth Server）**  
  负责 OAuth 2.1 / OIDC 授权、授权码交换、PKCE、JWKS、标准 token 与第三方 Client 管理。

### 2.2 设计原则

1. **协议标准化**：新接入尽量走 OAuth 2.1 / OIDC 授权码模式。
2. **体验一致性**：Account 仍作为统一入口，授权 UI 实现在 `/oauth/consent` 路径。
3. **渐进式迁移**：保留旧回跳方式作为兼容层，逐步迁移到 OAuth 标准。
4. **最小改造**：子应用接入仅需配置与最小代码片段，不引入重型 SDK。
5. **可观测性优先**：全链路记录授权行为与异常，利于回溯与治理。

## 3. 关键配置前提（Supabase OAuth Server）

该方案依赖 Supabase OAuth Server 功能配置：

- Site URL: `https://account.iasbt.com`
- Authorization Path: `/oauth/consent`
- Preview Authorization URL: `https://account.iasbt.com/oauth/consent`

该路径必须在 Account 项目中实现，用于 OAuth 授权 UI。此处是 OAuth Server 流程的核心入口，因此需要明确的用户授权体验与错误处理提示。

## 4. 新架构设计与核心流程图（文本化）

### 4.1 架构示意（文本版）

```
          ┌───────────────────────────┐
          │        子应用/第三方        │
          └─────────────┬─────────────┘
                        │
                        │ OAuth 2.1 Authorize 请求
                        │
          ┌─────────────▼─────────────┐
          │  Account 统一入口 (UI层)   │
          │  /oauth/consent 授权界面   │
          └─────────────┬─────────────┘
                        │
                        │ 授权确认
                        │
          ┌─────────────▼─────────────┐
          │  Supabase OAuth Server     │
          │  生成 code + PKCE 交换     │
          └─────────────┬─────────────┘
                        │
                        │ code -> token
                        │
          ┌─────────────▼─────────────┐
          │   子应用获取 token + session │
          └───────────────────────────┘
```

### 4.2 核心流程（授权码 + PKCE）

1. 子应用发起 OAuth authorize 请求，带 client_id、redirect_uri、code_challenge、scope 等参数。
2. Account 在 `/oauth/consent` 页面展示授权 UI（允许/拒绝）。
3. 用户授权后，Supabase OAuth Server 生成授权码（code）并回跳子应用。
4. 子应用用 code + code_verifier 交换 token（access_token、refresh_token）。
5. 子应用进入已登录状态，同时遵循 RLS 策略。

### 4.3 兼容流程（旧回跳 + 新 OAuth 并行）

旧模式：`/login?redirect=...`  
新模式：OAuth authorize + consent UI  
迁移方式：在接入向导中提供两套入口，旧模式作为兼容层，新模式作为推荐路线。

## 5. 数据模型与权限治理建议

### 5.1 数据模型补充

- **applications**：用于内部应用管理与向导输出，继续保留
- **oauth_clients**：由 Supabase OAuth Server 管理
- **user_app_access**：记录启动台访问行为，可扩展增加 oauth_client_id
- **auth_audit_log**（建议新增）：记录授权结果与失败原因（可先前端打点，后续后端入库）

### 5.2 权限治理

- 保持 profiles.role 权限模型与 AdminGuard
- OAuth scopes 与应用权限映射策略后续引入
- 为重要操作预留 MFA 或二次确认扩展点

## 6. 子应用接入策略

### 6.1 现有“懒人接入”继续可用

通过 Account 的 Applications 接入向导快速生成 `.env`、Login URL、AuthCallback 代码片段，维持原有低门槛接入体验。

### 6.2 新 OAuth 接入方式

为子应用提供 OAuth client 注册流程，强调：
- redirect URI 必须使用应用自身域名
- 使用授权码 + PKCE
- 失败回调与错误信息处理

### 6.3 渐进迁移机制

1. 新增应用默认使用 OAuth
2. 现有应用按优先级逐步迁移
3. 保留旧回跳模式至少一个版本周期

## 7. 用户体验与产品运营策略

### 7.1 用户体验

- 授权 UI 需明确“当前请求来源”
- 授权范围展示要清晰
- 授权失败要给出“返回上一页/重新授权”建议

### 7.2 运营与产品策略

- 通过启动台展示“已授权应用”列表
- 对授权使用频率与失败率进行分析，作为优化入口

## 8. 安全策略与异常处理

### 8.1 安全策略

- OAuth 授权码 + PKCE 防止 code 被截获
- 使用 JWKS 公开公钥，便于第三方验证
- 避免共享 JWT Secret，提升安全性

### 8.2 异常处理

- Consent UI 提示清晰失败原因
- 子应用在交换 token 失败时有回退路径
- 授权链路异常时记录事件（以便排查）

## 9. 测试与验收策略

### 9.1 单元测试

- 授权 UI 逻辑
- 回跳参数解析
- 接入向导输出逻辑

### 9.2 集成测试

- OAuth 授权码全链路
- Token 交换与 refresh

### 9.3 回归测试

- 保证旧回跳方式仍可使用
- 保证多浏览器一致

## 10. 运维与上线策略

### 10.1 灰度策略

- 先将 OAuth 模式应用限定为内部应用
- 逐步扩大到外部协作应用
- 保留旧入口一段时间

### 10.2 回滚策略

- OAuth 模式失败时回退至旧回跳模式
- 保留旧回跳入口与配置一段时间

  - 授权成功率与用户反馈优化

## 12. 可量化验收指标

- OAuth 授权成功率 ≥ 98%
- 授权失败原因可追踪率 ≥ 95%
- 旧回跳模式仍可用，回归测试全部通过
- 子应用接入步骤 ≤ 3 步（配置 + 按钮 + 回调页）

## 13. 结语

本计划书提出“双层架构”路线：Account 继续作为体验层，Supabase OAuth Server 作为协议层。该方案能够在保持现有产品体验的同时获得标准化授权能力，从而实现更稳健的安全性与可扩展性。通过清晰的里程碑与渐进式迁移策略，可以有效控制风险与开发成本，并为未来生态开放奠定基础。

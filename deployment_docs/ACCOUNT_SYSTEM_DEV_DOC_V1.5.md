## 4. 项目概况（约300字）

### 4.1 项目背景
Account System 是 Life OS 生态的统一身份中心，承担用户登录、权限控制与单点登录回跳。立项原因是多子应用重复建设认证系统导致体验割裂、权限口径不统一、运维成本居高不下。目标用户包括内部员工与生态子应用用户，核心业务痛点集中在“登录体验不一致”“权限散落难以审计”“跨应用跳转失效”。当前项目以统一入口为目标，提供可复用的 SSO 与 RBAC 能力，并作为子应用接入的统一基座。
为了保证小团队也能低成本维护，本项目强调轻量化部署与可观察性，避免引入复杂中间件。在保证权限一致的前提下，所有子应用统一从 Account 侧获取登录态或回跳 Token，提升跨应用的身份一致性与安全边界。该项目在功能完整性与运维复杂度之间做平衡选择，以“可维护”和“可演进”为核心原则。

### 4.2 项目范围（In / Out）
**In**
- 登录 / 登出 / 会话维护  
- RBAC 权限校验与管理端入口保护  
- SSO 回跳与白名单控制  
- 应用启动台与子应用接入配置  
- Nginx + PostgREST + Postgres 的轻量部署  

**Out**
- 订单中心与支付网关（旧方案已废弃）  
- Kubernetes / HPA（旧方案已废弃）  
- 分库分表与复杂分布式事务（旧方案已废弃）  
- 自研相册业务代码（已独立项目维护）  

### 4.3 关键指标
- 可用性：≥99.9%  
- 登录首屏可交互：P95 < 2s  
- SSO 回跳成功率：≥99.5%  
- 权限校验耗时：< 1ms（内存判定）  
- API 端点错误率：< 0.1%

## 5. 系统架构（约600字）

### 5.1 架构图
当前架构为三层分离：接入层（Nginx 网关 + 前端静态站点）、服务层（PostgREST 作为数据库 API）、数据层（PostgreSQL 15）。架构图源文件与可导出的 PNG 参考附件，分层展示请求从浏览器到网关、再到 API 与数据库的路径，并标注端口与数据流方向。
在数据流方面，浏览器访问前端页面时由 Nginx 提供静态资源；当页面需要获取用户信息或访问日志时，前端通过 /api/rest 进入 PostgREST，再由其执行 SQL 视图或表查询。所有内部服务通过 Docker 网络互通，减少对外暴露面，降低攻击面与端口被扫描的风险。

- 架构图源文件（Mermaid）：deployment_docs/attachments/architecture.mmd  
- 可编辑源文件（Draw.io）：deployment_docs/attachments/architecture.drawio  

### 5.2 技术栈
**前端**
- React 18 + Vite  
- TypeScript（严格模式）  
- TailwindCSS  
- Zustand 状态管理  
- React Router  

**后端与数据**
- PostgreSQL 15（容器化）  
- PostgREST（REST API 层）  
- Nginx 反向代理与静态托管  
整体架构避免引入额外中间件，减少版本漂移与维护成本。接口层由 PostgREST 直接映射数据库视图与表结构，降低 API 层代码维护负担，但对数据库建模与权限控制提出更高要求。

**版本锁定说明**
- PostgreSQL 15  
- Nginx alpine 镜像  
- PostgREST latest（通过镜像管理版本更新）  

### 5.3 部署模型
**现状部署**
- 单机 Docker Compose：Nginx、PostgREST、Postgres 三容器同网络  
- 内部网络互通，仅 Nginx 对外暴露 80/443/8080  
- PostgREST 与 Postgres 仅内网可见  
该部署模式强调“可迁移”，通过 deploy_to_remote.ps1 脚本实现一键部署，适合低成本环境迁移与快速恢复。运维策略以“保守可靠”为主，不追求过度自动化，避免因复杂流水线导致不可控风险。

**旧方案（已废弃）Kubernetes + Docker**
- 副本数：Nginx 2、PostgREST 2、Postgres 1  
- 资源限制：每容器 0.5 CPU / 512Mi，数据库 2 CPU / 4Gi  
- HPA：CPU 70% 扩容阈值，最大 4 副本  
该方案未在当前项目启用，仅作为历史设计保留。

## 6. 核心模块设计（约800字）

### 6.1 用户管理
当前账号系统的核心能力是登录与权限控制。登录流程由前端发起，完成鉴权后写入本地会话状态，并在访问受限路由时通过 RBAC 判定。权限模型以角色为中心，常见角色为 user 与 admin，路由守卫仅在用户具有 admin 时放行管理端。SSO 回跳采用 redirect 参数模式，登录成功后优先回跳至来源应用。当前项目仍保留 Supabase 认证模型的对接路径，作为历史迁移方案存在；若未来迁移到自建认证服务，可在 useAuthStore 中替换 session 的来源与校验逻辑。
为避免冷启动导致登录卡顿，初始化逻辑采用并行查询与超时兜底机制，确保 2 秒内没有响应时直接给出最小权限，防止白屏。退出登录采用强制硬刷新策略，保证移动端浏览器缓存被彻底清理，避免“登录-退出”循环问题。该设计保证了体验稳定，但也要求对缓存更新和版本号进行严格管理。
权限模型中，视图与操作权限以统一的权限标识进行归类，前端通过 hasPermission 判定是否展示入口，路由层级通过 AdminGuard 进行拦截，确保权限无法被绕过。为降低接入复杂度，子应用只需关注回跳参数与 allowlist 校验，无需感知账号系统内部权限细节。

**密码加密策略（规范要求）**  
旧文档约定使用 Argon2id + 随机 salt 进行密码哈希，并将参数版本化存储。当前项目未落地自建密码库，若启用自建认证服务，需统一盐值生成与校验迭代策略。

**OAuth2 集成（旧方案已废弃）**  
旧方案拟将第三方应用接入 OAuth2，当前未启用，仅保留接口契约与配置示例。

### 6.2 订单中心（已废弃旧方案）
订单中心从未在当前项目实现，历史方案仅用于未来扩展讨论。旧方案规划包含状态机与幂等性设计：订单状态包括 CREATED、PAID、CANCELLED、REFUNDED；每次支付回调通过幂等键校验避免重复扣款；跨服务事务采用 Saga 模式，先冻结库存再确认支付，失败则补偿回滚。该模块与当前账号系统无直接关系，故在 1.5 版本中标注为已废弃。
在旧方案中，状态流转严格限制跳转路径，并要求所有写操作记录业务审计日志，便于后续对账与风控复盘。该部分仅作为历史设计备查，不纳入当前实施范围。

### 6.3 支付网关（已废弃旧方案）
支付网关为历史设想组件，包含渠道路由与对账解析。设计上根据用户地域和渠道健康状态选择支付路径，失败可按指数退避重试，对账文件解析采用定时任务入库并对账异常触发人工复核。当前项目未包含支付链路，故该段仅作为废弃说明保留。
旧方案中风控规则包含单账户频次、异常支付金额与黑名单拦截，配合风控日志形成闭环。该能力不在当前项目范围内。

### 6.4 接口契约
本项目对外接口主要由 PostgREST 提供，统一由 Nginx 代理至 /api/rest。接口契约在附件提供 OpenAPI 3.0 YAML 文件，包含统一错误码、分页参数规范与基础资源查询示例。
契约强调前后端一致性，所有分页接口统一以 page / page_size 作为入参，返回结构中保持字段命名与数据类型一致，避免前端逻辑分支爆炸。错误码采用五位编码，首位表示模块，第二位表示业务域，后三位为具体错误编号，便于快速定位故障来源。

- OpenAPI 3.0：deployment_docs/attachments/openapi.yaml  
- 错误码规范：5 位数字，首位表示模块，如 10001（认证）、20001（权限）、30001（应用）  
- 分页参数：page / page_size，最大 page_size = 100

## 7. 数据模型（约400字）

### 7.1 ER 图
当前数据库以 profiles 与 user_app_access 为核心表，用于用户档案与访问审计；applications 表用于子应用接入配置。ER 图使用 dbdiagram.io 的 DBML 描述，见附件，可生成可视化 ER 图并标注主键、外键、索引与唯一约束。
建议对 applications.code 建立唯一索引以保证接入配置唯一，对 user_app_access.user_id 建立普通索引提升查询性能。外键约束用来维持访问日志与用户、应用之间的完整性。

- ER 源码：deployment_docs/attachments/dbdiagram.dbml  

### 7.2 字段说明
规范要求每个表新增创建时间、更新时间与逻辑删除字段（created_at / updated_at / deleted_at）。当前数据库以最小字段集运行为主，若启用审计或数据清理，建议补齐字段并建立索引。枚举值需配套字典表，如 role 字典表、app_status 字典表，以便跨模块一致读取。
为保证数据一致性，建议将用户角色、应用状态等枚举值独立管理，并在业务逻辑中只通过字典表映射读取。索引策略建议以 user_id 与 app_id 为主，优先保证访问日志写入与查询的性能稳定。

### 7.3 分库分表策略（已废弃旧方案）
旧方案按 user_id 哈希分 16 库 64 表，提供扩容脚本与回滚方案。当前项目未启用分库分表，保留此段为历史规划说明。扩容脚本与回滚方案未执行，不作为现行运维要求。

## 8. 接口文档（约500字）

### 8.1 鉴权方式
旧方案定义 JWT + Refresh Token 机制，访问令牌 30min、刷新令牌 7d、刷新窗口 5min。当前项目主要以会话态与内网 API 访问为主，JWT 机制未落地，仅保留规范定义。
如后续启用自建认证服务，JWT 需要包含 user_id、role、exp 等字段，刷新令牌需写入持久化存储并支持主动失效，确保异常登录可随时吊销会话。
为降低安全风险，建议引入统一 allowlist 校验并限制回跳域名。回跳 URL 应经过严格校验后再执行跳转，避免开放重定向。

### 8.2 签名规则
旧方案定义时间戳 + Nonce + Body 排序后 HMAC-SHA256，防重放有效期 60s。当前项目未启用签名网关，保留该规范作为未来扩展。
签名方案要求所有客户端请求带上 nonce 与 timestamp，服务端需维护短期缓存并拒绝重复 nonce，避免重放攻击。

### 8.3 示例请求
以下示例以 /api/rest/profiles 为例，展示请求与返回字段含义。  

**curl**
```bash
curl -X GET "https://www.iasbt.cloud/api/rest/profiles?page=1&page_size=20"
```

**JavaScript**
```javascript
fetch("https://www.iasbt.cloud/api/rest/profiles?page=1&page_size=20", {
  method: "GET"
}).then(res => res.json())
```

**Python**
```python
import requests
resp = requests.get("https://www.iasbt.cloud/api/rest/profiles?page=1&page_size=20")
print(resp.json())
```

**返回字段说明**
- id：用户主键 UUID  
- email：邮箱  
- role：角色，取值 user / admin  
- avatar_url：头像 URL  
错误返回采用统一结构，code 为五位数字，message 为人类可读信息。分页接口默认返回数组，需要时可由前端追加总数查询。

## 9. 安全与合规（约200字）

### 9.1 OWASP Top 10 对应防护
SQL 注入：仅通过 PostgREST 与参数化查询访问数据库。  
XSS：前端统一转义用户输入，禁止危险 HTML。  
CSRF：管理端接口保持同源策略并限制跨域。  
弱口令：旧方案要求密码复杂度，当前未启用自建密码库。  
敏感数据泄露：日志脱敏与最小权限访问。  
在当前架构中，Nginx 作为唯一入口可统一落地安全策略，如请求大小限制、基础限流与黑名单拦截。对于开放重定向风险，建议使用 allowlist 校验回跳域名，防止被恶意利用。

### 9.2 隐私合规
遵循 GDPR/CCPA 的数据主体权利（访问、更正、删除）。跨境传输需进行评估与备案。敏感数据建议使用 AES-256-GCM 加密存储，密钥统一管理。
传输层统一使用 HTTPS，严禁在日志与监控中记录明文密钥。对访问审计数据应设置最小可见范围，防止内部越权。

## 12. 上线与运维（约100字）

### 12.1 CI/CD
旧方案采用 GitLab Runner，stage 包含 lint、test、build、scan、deploy。当前项目以脚本部署为主，CI/CD 方案保留为模板。

### 12.2 监控告警
Prometheus + Grafana 监控核心接口 P99 延迟、错误率、Pod 重启次数；当前环境未部署监控，作为运维规范要求。

### 12.3 灾备
每日冷备、每小时 Binlog 增量备份，RPO ≤ 15min、RTO ≤ 30min，为目标规范。
建议在备份流程中加入完整性校验与恢复演练，确保数据可用性与灾备方案可落地。

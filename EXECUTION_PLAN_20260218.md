# 后续实施方案 (2026-02-18)

## 1. 概述
鉴于我们已成功完成数据库私有化迁移和基础架构搭建，本方案旨在规划未来 3-6 个月的系统演进路线。重点聚焦于**安全性加固**、**运维自动化**、**业务功能扩展**以及**高可用架构升级**四个维度，确保系统在承载更多用户和数据时依然稳健可靠。

## 2. 第一阶段：安全加固与基础运维 (T+1周)

### 2.1 数据库安全加固
*   **目标**: 消除当前部署中的安全隐患。
*   **行动**:
    1.  **关闭公网端口**: 修改 `docker-compose.yml`，移除 `postgres-business` 的 `5432:5432` 映射，仅保留容器间通信。外部管理通过 SSH 隧道进行。
    2.  **密码轮换**: 更新数据库 root 密码，并使用 Docker Secret 或环境变量文件 (`.env`) 进行更严格的权限控制，避免明文密码出现在脚本中。
    3.  **网络隔离**: 审查 Docker 网络配置，确保数据库容器无法直接访问公网（Egress Filtering），仅允许必要的更新源访问。

### 2.2 自动化备份体系
*   **目标**: 建立 RPO (恢复点目标) < 24小时的灾备能力。
*   **行动**:
    1.  **脚本开发**: 编写 Shell 脚本，每日凌晨 3 点自动执行 `pg_dump` 和 `mysqldump`。
    2.  **异地存储**: 配置腾讯云 COS 或阿里云 OSS CLI 工具，将压缩后的备份文件上传至云端存储桶。
    3.  **保留策略**: 实施 "7天日备 + 4周周备 + 6月月备" 的保留策略。
    4.  **恢复演练**: 每月执行一次从云端备份恢复到测试环境的演练，验证备份文件的可用性。

### 2.3 HTTPS 全面覆盖
*   **目标**: 提升通信安全性，避免浏览器“不安全”警告。
*   **行动**:
    1.  **证书申请**: 为 `*.iasbt.cloud` 申请通配符 SSL 证书（Let's Encrypt 自动续期）。
    2.  **Nginx 配置**: 强制开启 HTTP -> HTTPS 重定向 (HSTS)。
    3.  **安全头部**: 配置 `Strict-Transport-Security`, `X-Content-Type-Options`, `X-Frame-Options` 等安全响应头。

## 3. 第二阶段：前端功能深化与体验优化 (T+2周)

### 3.1 账户中心功能完善
*   **目标**: 让用户能够自主管理其数据。
*   **行动**:
    1.  **个人资料编辑**: 基于 PostgREST API 开发用户资料修改页面（头像、昵称、简介）。
    2.  **安全中心**: 集成 Casdoor SDK，提供修改密码、绑定邮箱/手机号、查看登录日志的功能。
    3.  **应用授权管理**: 展示用户已授权的应用列表，允许用户主动解除 OAuth 授权。

### 3.2 数据可视化仪表盘
*   **目标**: 直观展示系统运行状态。
*   **行动**:
    1.  **实时统计**: 在 Dashboard 展示当前在线用户数、今日新增照片数（对接 MT Photos API）、存储空间使用率。
    2.  **图表集成**: 使用 Recharts 或 ECharts 绘制近 30 天的登录活跃度趋势图。

## 4. 第三阶段：CI/CD 与监控体系 (T+1个月)

### 4.1 持续集成/持续部署 (CI/CD)
*   **目标**: 实现代码提交即自动部署，减少人工操作风险。
*   **行动**:
    1.  **GitHub Actions**: 配置 Workflow，当 `main` 分支有代码推送到 GitHub 时：
        *   自动运行单元测试 (`npm test`)。
        *   自动构建 Docker 镜像并推送到私有仓库。
        *   通过 SSH 触发服务器更新服务 (`docker compose pull && docker compose up -d`)。

### 4.2 全链路监控
*   **目标**: 变“被动救火”为“主动预防”。
*   **行动**:
    1.  **基础监控**: 部署 Node Exporter + Prometheus，采集服务器 CPU、内存、磁盘、网络指标。
    2.  **容器监控**: 使用 cAdvisor 采集各个 Docker 容器的资源使用情况。
    3.  **可视化**: 搭建 Grafana 看板，配置 CPU 使用率 > 80% 或磁盘剩余空间 < 10% 的告警通知（通过企业微信或邮件）。
    4.  **日志聚合**: 考虑引入 Loki 或 ELK Stack，统一收集 Nginx、Casdoor 和业务服务的日志，便于快速排查 502/500 错误。

## 5. 第四阶段：高可用与性能优化 (T+3个月)

### 5.1 数据库性能调优
*   **目标**: 应对未来数据量增长（预计 10万+ 用户）。
*   **行动**:
    1.  **索引优化**: 根据慢查询日志，对高频查询字段（如 `profiles.username`, `user_consent_logs.user_id`）添加合适的 B-Tree 或 Hash 索引。
    2.  **参数调优**: 根据服务器内存（4GB/8GB），调整 PostgreSQL 的 `shared_buffers`, `work_mem`, `effective_cache_size` 等关键参数。

### 5.2 服务高可用
*   **目标**: 消除单点故障。
*   **行动**:
    1.  **负载均衡**: 在前端增加一层 Cloud Load Balancer (CLB)。
    2.  **多节点部署**: 如果业务量激增，考虑将数据库独立部署到专用实例，应用服务扩展为多节点集群模式。

## 6. 结语
本实施方案遵循“先稳后快、步步为营”的原则。当前首要任务是夯实基础（安全与备份），随后逐步丰富应用功能，最后构建自动化与监控体系以支撑长期发展。我们将严格按照此路线图推进，确保 Account 项目成为一个成熟、企业级的身份与数据中心。
